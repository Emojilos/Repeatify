# PRD: Repeatify — Платформа для подготовки к ЕГЭ по профильной математике

**Версия:** 1.0 (MVP)
**Дата:** 2026-03-06
**Статус:** Draft

---

## 1. Обзор продукта и цели

### 1.1 Описание

Repeatify — веб-платформа для подготовки к ЕГЭ по профильной математике, построенная на научно обоснованных методах обучения. Ключевое отличие от существующих решений (Решу ЕГЭ, Умскул, ФИПИ) — интеграция адаптивного интервального повторения (Spaced Repetition) с иерархическим графом математических знаний, что обеспечивает максимальную эффективность обучения при минимальных временных затратах.

### 1.2 Проблема

Существующие платформы подготовки к ЕГЭ обладают критическими недостатками:
- **ФИПИ:** устаревшая архитектура, отсутствие адаптивности и трекинга прогресса
- **Решу ЕГЭ:** нерелевантный авторский контент, устаревший UI, отсутствие персонализации
- **Онлайн-школы (Умскул, Фоксфорд):** линейные видеолекции, неадаптивные фиксированные интервалы повторения, не учитывают индивидуальную кривую забывания

Абитуриенты вынуждены комбинировать разрозненные инструменты и вручную отслеживать прогресс.

### 1.3 Решение

Centrализованная платформа, объединяющая:
- Адаптивный алгоритм интервального повторения (FSRS) с персонализацией
- Иерархический граф знаний с моделью FIRe (Fractional Implicit Repetition) для компрессии повторений
- Адаптивные учебные планы на основе оставшегося времени до экзамена
- Пошаговое раскрытие решений для задач второй части
- Систему мотивации (стрики + тепловая карта активности)

### 1.4 Целевая аудитория

- **Первичная:** учащиеся 10–11 классов, готовящиеся к ЕГЭ по профильной математике (возраст 15–18 лет)
- **Вторичная:** выпускники прошлых лет, пересдающие экзамен; родители, отслеживающие прогресс

### 1.5 Ключевые метрики успеха (KPI)

| Метрика | Целевое значение |
|---------|------------------|
| DAU/MAU (stickiness) | > 40% |
| Средняя длина стрика | > 7 дней |
| Retention Day-7 | > 50% |
| Среднее время сессии | 15–30 минут |
| Completion rate ежедневной тренировки | > 70% |

### 2.2 Frontend

| Технология | Назначение |
|-----------|------------|
| React 18+ | SPA-фреймворк |
| React Router | Маршрутизация |
| Tailwind CSS | Стилизация |
| KaTeX | Рендеринг математических формул (LaTeX) |
| Zustand или React Context | State management |
| Supabase JS Client | Взаимодействие с БД и Auth |
| Vite | Сборка и dev-server |

**Почему React + GitHub Pages:** React SPA отлично работает как статический сайт на GitHub Pages. Вся логика маршрутизации на клиенте. Для GitHub Pages SPA необходим `404.html` → `index.html` redirect trick или HashRouter.

**Почему KaTeX (а не MathJax):** KaTeX рендерит формулы в 10–100 раз быстрее MathJax, что критично для интерфейса с большим количеством математических выражений на экране.

### 2.3 Backend

| Технология | Назначение |
|-----------|------------|
| Python 3.11+ / FastAPI | REST API сервер |
| FSRS (py-fsrs) | Алгоритм интервального повторения |
| NetworkX | Граф знаний (Knowledge Graph) |
| Selenium + BeautifulSoup | Парсинг заданий ФИПИ |
| MinerU | OCR и парсинг PDF |
| Хостинг: Render (Free/Starter) | Деплой бэкенда |

**Альтернатива:** Если бэкенд-логика минимальна на MVP, можно реализовать FSRS-алгоритм на клиенте (JavaScript) и использовать Supabase Edge Functions для серверной логики. Это устранит необходимость в Render на начальном этапе.

### 2.4 База данных и инфраструктура

| Сервис | Назначение | Тир |
|--------|-----------|-----|
| Supabase | PostgreSQL БД + Auth + Storage + Edge Functions | Free (500MB БД, 50K MAU Auth) |
| GitHub Pages | Хостинг фронтенда | Free |
| Render | Хостинг бэкенда (Python) | Free (spin-down after 15 min inactivity) |

**Важное ограничение Render Free:** сервер засыпает после 15 минут неактивности. Первый запрос после пробуждения может занять 30–60 секунд. Решение: использовать cron-job для периодического пинга или перейти на Starter ($7/мес) при росте аудитории.

### 3.2 Row Level Security (RLS) — Supabase

Все таблицы с пользовательскими данными должны иметь RLS-политики:

```sql
-- Пример для user_card_progress
ALTER TABLE user_card_progress ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users can view own progress"
  ON user_card_progress FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can insert own progress"
  ON user_card_progress FOR INSERT
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update own progress"
  ON user_card_progress FOR UPDATE
  USING (auth.uid() = user_id);
```

Аналогичные политики для: `users`, `user_topic_mastery`, `study_sessions`, `review_logs`, `daily_activity`.

Таблицы `topics`, `topic_dependencies`, `cards` — доступны на чтение всем аутентифицированным пользователям, запись только через admin/service role.

## 4. Основные функции (MVP)

### 4.1 Функция: Аутентификация и онбординг

**Описание:** Регистрация и вход по email + пароль через Supabase Auth. После регистрации — онбординг для настройки учебного плана.

**User Flow:**
1. Пользователь заходит на сайт → видит лендинг
2. Нажимает «Начать подготовку» → форма регистрации (email + пароль)
3. Подтверждает email (Supabase magic link)
4. Проходит онбординг:
   - Шаг 1: «Когда у тебя экзамен?» → выбор даты
   - Шаг 2: «Сколько минут в день готов заниматься?» → 15 / 30 / 45 / 60
   - Шаг 3: «Какой у тебя текущий уровень?» → диагностический мини-тест (5–10 задач разной сложности)
5. Система генерирует персональный учебный план → переход в личный кабинет

**Acceptance Criteria:**
- [ ] Регистрация через email/пароль работает корректно
- [ ] Email-подтверждение отправляется и обрабатывается
- [ ] Вход/выход работает, сессия сохраняется
- [ ] Онбординг собирает: дату экзамена, ежедневную цель, начальный уровень
- [ ] Диагностический тест корректно определяет начальный уровень по кодификатору
- [ ] После онбординга создаётся запись в `users` с заполненными полями `exam_date`, `daily_goal_minutes`, `study_plan_type`

**Technical Considerations:**
- Supabase Auth JS SDK: `@supabase/supabase-js`
- Хранение JWT токена в памяти (не localStorage для безопасности, Supabase SDK управляет этим автоматически)
- Защита роутов React через AuthContext + PrivateRoute компонент

### 4.2 Функция: Граф знаний (Knowledge Graph) и модель FIRe

**Описание:** Иерархическая структура всех тем профильного ЕГЭ, представленная как направленный ациклический граф (DAG). Каждый узел — математический навык, рёбра — связи «базовый → продвинутый» с весами включения.

**Структура графа (укрупнённая):**
```
Профильный ЕГЭ по математике
├── Алгебра
│   ├── Числа и вычисления
│   ├── Алгебраические выражения
│   ├── Уравнения
│   │   ├── Линейные уравнения [weight=1.0 → Квадратные уравнения]
│   │   ├── Квадратные уравнения [weight=0.8 → Показательные уравнения]
│   │   ├── Показательные уравнения
│   │   ├── Логарифмические уравнения [weight=0.7 → Лог. неравенства]
│   │   └── Тригонометрические уравнения
│   ├── Неравенства
│   ├── Функции и графики
│   └── Производная и первообразная
├── Геометрия
│   ├── Планиметрия (задание 16)
│   └── Стереометрия (задание 14)
├── Начала мат. анализа
├── Теория вероятностей и статистика
├── Комбинаторика
└── Задачи с параметром (задание 18)
```

**Механизм FIRe (Fractional Implicit Repetition):**

1. **Нисходящий поток кредита (trickle-down credit):** когда ученик успешно решает задачу продвинутого уровня, система начисляет неявное повторение (implicit credit) всем базовым темам-пререквизитам. Кредит пропорционален весу ребра (`weight`).

2. **Восходящий поток штрафов (penalty flow):** при ошибке на базовой теме система блокирует или увеличивает частоту повторения зависимых продвинутых тем.

3. **Компрессия повторений:** вместо 10 отдельных карточек на базовые темы алгоритм подбирает 2–3 комплексные задачи, которые неявно покрывают все необходимые повторения.

**Acceptance Criteria:**
- [ ] Граф знаний загружен в БД и покрывает все задания 1–19 ЕГЭ
- [ ] Каждая тема имеет связи (prerequisite → dependent) с весами
- [ ] При успешном решении продвинутой задачи пересчитывается `implicit_credit` для базовых тем
- [ ] При ошибке на базовой теме повышается приоритет повторения зависимых тем
- [ ] `user_topic_mastery` обновляется после каждой сессии

**Pseudocode: Распространение кредита FIRe**
```python
def propagate_credit(user_id, card_id, rating, graph):
    """
    После успешного ответа (rating >= 3) на продвинутую задачу,
    распространяет неявный кредит по графу знаний вниз.
    """
    card = get_card(card_id)
    topic = get_topic(card.topic_id)

    if rating < 3:  # Ошибка — распространяем штраф вверх
        propagate_penalty_up(user_id, topic, graph)
        return

    # Получаем все пререквизиты (рёбра вниз)
    prerequisites = graph.get_prerequisites(topic.id)

    for prereq_edge in prerequisites:
        base_topic = prereq_edge.prerequisite_topic
        weight = prereq_edge.weight  # 0.0–1.0

        mastery = get_user_topic_mastery(user_id, base_topic.id)

        # Дисконтирование: если последнее явное повторение было < 1 дня назад,
        # неявный кредит уменьшается
        days_since_explicit = days_between(mastery.last_explicit_review, now())
        discount = min(1.0, days_since_explicit / 1.0)

        credit = weight * discount
        mastery.implicit_credit += credit
        mastery.last_implicit_review = now()

        # Если implicit_credit достаточен, обновляем FSRS-интервалы
        # для карточек этой темы
        if mastery.implicit_credit >= 1.0:
            extend_intervals_for_topic(user_id, base_topic.id)
            mastery.implicit_credit -= 1.0

        save(mastery)

        # Рекурсивное распространение с затуханием
        if weight > 0.3:  # Порог затухания
            propagate_credit(user_id, card_id, rating, graph,
                           accumulated_weight=weight * 0.7)
```

### 4.3 Функция: Spaced Repetition Engine (FSRS)

**Описание:** Ядро платформы — алгоритм FSRS (Free Spaced Repetition Scheduler), определяющий оптимальное время показа каждой карточки для каждого ученика.

**Модель памяти DSR (Difficulty, Stability, Retrievability):**
- **Difficulty (D):** объективная сложность карточки для данного ученика (0–10)
- **Stability (S):** время в днях, за которое вероятность вспоминания падает до 90%
- **Retrievability (R):** текущая вероятность успешного вспоминания (0–1)

**Шкала оценки ответа:**
- **1 (Again):** полное незнание — вернуть карточку на ближайший показ
- **2 (Hard):** ответ с большим трудом
- **3 (Good):** правильный ответ с нормальным усилием
- **4 (Easy):** мгновенный правильный ответ

**Реализация:**
- Использовать open-source библиотеку `py-fsrs` (Python) или `ts-fsrs` (TypeScript/JS)
- MVP: начать с дефолтных параметров FSRS
- v2: обучать персональные параметры на логах `review_logs` конкретного ученика

**Acceptance Criteria:**
- [ ] FSRS-алгоритм корректно рассчитывает `due_date` для каждой карточки после ответа
- [ ] Параметры S, D, R сохраняются в `user_card_progress`
- [ ] Каждый ответ логируется в `review_logs` с `rating`, `response_time_ms`, `hints_used`
- [ ] Новые карточки показываются в порядке, определённом графом знаний (от простых к сложным)
- [ ] Ежедневная сессия формируется из: карточки с `due_date <= now()` + новые карточки (лимит зависит от `daily_goal_minutes`)
- [ ] Интерливинг: карточки в сессии перемешаны по темам (не более 2 подряд из одной темы)

**Формирование ежедневной сессии (pseudocode):**
```python
def generate_daily_session(user_id):
    """
    Генерирует оптимальную ежедневную тренировочную сессию.
    """
    user = get_user(user_id)
    time_budget = user.daily_goal_minutes

    # 1. Обязательные: карточки с истекшим сроком (due)
    due_cards = get_due_cards(user_id, limit=50)
    due_cards = sort_by_priority(due_cards)  # overdue + низкий retrievability первыми

    # 2. Компрессия через FIRe: заменяем простые карточки комплексными
    compressed = apply_fire_compression(user_id, due_cards)

    # 3. Новые карточки (если осталось время)
    remaining_time = time_budget - estimate_time(compressed)
    if remaining_time > 5:
        new_cards = get_next_new_cards(user_id,
                                       time_budget=remaining_time,
                                       graph=knowledge_graph)

    # 4. Интерливинг: перемешиваем по темам
    session = interleave(compressed + new_cards, max_consecutive_same_topic=2)

    return session
```

### 4.4 Функция: Адаптивный учебный план (Study Plan)

**Описание:** Система адаптирует интенсивность и приоритеты обучения на основе оставшегося времени до экзамена.

**Режимы обучения:**

| Режим | Время до экзамена | Фокус | Новые карточки/день | Описание |
|-------|-------------------|-------|---------------------|----------|
| `relaxed` | > 9 месяцев | Фундамент | 5–10 | Медленное, глубокое изучение. Акцент на понимание базовых концепций |
| `standard` | 4–9 месяцев | Баланс | 10–20 | Равномерное продвижение по всем темам. Введение задач повышенной сложности |
| `intensive` | 1–4 месяца | Слабые зоны | 15–25 | Приоритет тем с низким mastery. Увеличение доли задач второй части |
| `sprint` | < 1 месяца | Удержание | 5–10 новых, 30+ повторений | Минимум нового материала. Максимум повторений. Фокус на задания, дающие максимум баллов |

**Логика адаптации:**
1. Ежедневно система пересчитывает `days_until_exam`
2. На основе `days_until_exam` и текущего `mastery_level` корректирует:
   - Количество новых карточек в день
   - Приоритет тем (в sprint-режиме — задания 1–12 для гарантированных баллов)
   - Желаемый уровень retention (в sprint — 95%, в relaxed — 85%)
3. В sprint-режиме система рекомендует фокусироваться на задачах с максимальным ROI (баллы/усилие)

**Acceptance Criteria:**
- [ ] При онбординге пользователь устанавливает дату экзамена
- [ ] Система автоматически определяет режим обучения и пересчитывает его ежедневно
- [ ] Количество новых карточек в день соответствует режиму
- [ ] В sprint-режиме приоритизируются задания 1–12 (часть 1) для максимизации баллов
- [ ] Пользователь может изменить дату экзамена и ежедневную цель в настройках
- [ ] В личном кабинете отображается текущий режим и оставшееся время

### 4.5 Функция: Пошаговое раскрытие решений (Step-by-Step)

**Описание:** Для задач второй части ЕГЭ (задания 13–19) решения разбиваются на логические шаги с педагогическими комментариями. Каждый шаг можно раскрывать отдельно.

**Формат данных `solution_steps` (JSONB):**
```json
[
  {
    "step_number": 1,
    "title": "Определяем тип уравнения",
    "text": "Перед нами логарифмическое уравнение с переменным основанием. Применяем метод рационализации.",
    "latex": "\\log_{x-1}(2x+3) = 2",
    "hint": "Обратите внимание на ОДЗ: основание логарифма должно быть > 0 и ≠ 1"
  },
  {
    "step_number": 2,
    "title": "Находим ОДЗ",
    "text": "Записываем систему ограничений на область допустимых значений.",
    "latex": "\\begin{cases} x - 1 > 0 \\\\ x - 1 \\neq 1 \\\\ 2x + 3 > 0 \\end{cases}",
    "hint": null
  },
  {
    "step_number": 3,
    "title": "Решаем уравнение",
    "text": "Используем определение логарифма для перехода к показательной форме.",
    "latex": "(x-1)^2 = 2x + 3",
    "hint": "По определению: если log_a(b) = c, то a^c = b"
  }
]
```

**UI-паттерн:**
1. Пользователь видит условие задачи
2. Пытается решить самостоятельно
3. Может нажать «Показать шаг 1» → раскрывается первый шаг с анимацией
4. После каждого шага — кнопки «Понятно, дальше решу сам» / «Показать следующий шаг»
5. Если пользователь использовал подсказки/шаги, это влияет на FSRS-рейтинг:
   - 0 подсказок → пользователь оценивает сам (Again/Hard/Good/Easy)
   - 1–2 подсказки → максимальная оценка ограничена до Hard (2)
   - Все шаги раскрыты → автоматически Again (1), карточка возвращается

**Acceptance Criteria:**
- [ ] Для карточек типа `step_by_step` решение отображается поэтапно
- [ ] Каждый шаг содержит текст, LaTeX-формулу и опциональную подсказку
- [ ] LaTeX рендерится через KaTeX корректно на каждом шаге
- [ ] Количество использованных подсказок/шагов записывается в `review_logs.hints_used`
- [ ] Использование подсказок автоматически влияет на максимальный FSRS-рейтинг
- [ ] Анимация раскрытия шагов плавная (CSS transition, 300ms)
- [ ] Работает корректно на мобильных экранах

- Общее количество решённых карточек
- Средняя точность ответов (%)
- Время, потраченное на обучение (за неделю/месяц)

- [ ] Стрик рассчитывается корректно (учёт часовых поясов пользователя)

- [ ] Адаптивный дизайн: корректно отображается на экранах 320px–1920px

### 4.7 Функция: Система стриков и мотивации

**Описание:** Система мотивации, основанная на поддержании непрерывной серии дней обучения (streak) и визуализации прогресса.

**Механика стрика:**
1. День засчитывается для стрика, если пользователь завершил минимум 50% ежедневной сессии
2. При пропуске дня стрик обнуляется
3. Streak Freeze: 1 заморозка в неделю (автоматически применяется при пропуске)
4. Отображение: текущий стрик + рекорд (longest streak)
5. Уведомления (в UI, не push): «Не забудь позаниматься! Твой стрик: 7 дней 🔥»

**Визуальные элементы мотивации:**
- Тепловая карта активности (описана в 4.6.2)
- Анимация при достижении milestone стриков (7, 14, 30, 60, 100 дней)
- Счётчик обратного отсчёта до экзамена на Dashboard
- Прогресс-бар общей готовности к экзамену (% освоения всех тем)

**Calm UX принципы:**
- Нет агрессивных звуковых сигналов при ошибке
- Ошибочный ответ → конструктивная обратная связь: «Близко! Давай разберём этот момент»
- Нет жёстких таймеров в тренировочных сессиях (только в режиме пробного ЕГЭ)
- Сообщения в стиле Growth Mindset: похвала за упорство, а не за талант

**Acceptance Criteria:**
- [ ] Стрик корректно увеличивается при выполнении ≥ 50% сессии
- [ ] Стрик обнуляется при пропуске дня (с учётом Streak Freeze)
- [ ] 1 Streak Freeze в неделю, применяется автоматически
- [ ] Тепловая карта обновляется в реальном времени после завершения сессии
- [ ] Milestone-анимации работают на 7, 14, 30, 60, 100 днях
- [ ] Обратная связь при ошибке — конструктивная, без стигматизации
- [ ] Нет таймеров в обычных тренировочных сессиях

## 5. Контентная стратегия и pipeline данных

### 5.1 Источники контента для MVP

#### Фаза 1: Ручная база + полуавтоматический парсинг (запуск)
1. **Ручная база (первый приоритет):** 50–100 отобранных задач по ключевым темам, размеченных вручную с solution_steps и привязкой к графу знаний
2. **Парсинг ФИПИ:** автоматизированный сбор задач из открытого банка ФИПИ через Selenium + BeautifulSoup (архитектура описана в проектном документе)

#### Фаза 2: Расширение (после запуска)
3. **PDF-импорт:** парсинг сборников и пробных вариантов через MinerU / OCR
4. **AI-генерация вариативных задач:** на основе существующих задач менять коэффициенты и параметры для создания новых вариантов

### 5.2 Pipeline парсинга ФИПИ

```
[ФИПИ Web] → [Selenium WebDriver (headless Chrome)]
            → [Explicit Wait для рендеринга]
            → [BeautifulSoup: извлечение HTML]
            → [Парсер: текст + LaTeX + изображения]
            → [Классификация по кодификатору ЕГЭ]
            → [Валидация контента]
            → [Запись в Supabase: таблица cards]
```

**Технические требования к парсеру:**
- Поддержка динамического рендеринга (Client-Side Rendering сайта ФИПИ)
- Ротация User-Agent и задержки между запросами (2–5 секунд)
- Извлечение: текст условия, варианты ответов, изображения, метаданные (тема, сложность)
- Конвертация математической нотации в LaTeX
- Сохранение изображений (чертежи, графики) в Supabase Storage

### 5.3 Формат хранения математического контента

Все математические выражения хранятся в формате LaTeX:
- Inline: `$x^2 + 2x + 1 = 0$`
- Block: `$$\int_0^1 f(x)\,dx = F(1) - F(0)$$`
- Рендеринг на клиенте: KaTeX с авто-обнаружением `$...$` и `$$...$$`

| Render Free spin-down (30–60 сек на первый запрос) | Высокий | Cron-пинг каждые 14 мин; или перенести FSRS-логику на клиент + Supabase Edge Functions |
| Парсинг ФИПИ: изменение вёрстки сайта | Средний | Модульный парсер с отделёнными CSS-селекторами; мониторинг и алерты при сбоях |
| Конфликт LaTeX-синтаксиса в карточках | Средний | Использовать KaTeX safe mode; экранировать спецсимволы при сохранении |
| Холодный старт FSRS (нет данных о пользователе) | Низкий | Использовать дефолтные параметры FSRS + результаты диагностического теста для начальной калибровки |
| Supabase Free tier лимиты (500MB, 2GB transfer) | Средний при росте | Мониторинг использования; план миграции на Pro ($25/мес) при достижении 70% лимита |
| Мотивация пользователей: отток после 2–3 недель | Высокий | Стрики + тепловая карта + Calm UX + обратная связь о прогрессе; email-напоминания (v2) |
| Сложность реализации FIRe-модели | Средний | MVP: упрощённая версия (только полные включения weight=1.0); v2: полная модель с частичными включениями |

## 12. Глоссарий

| Термин | Определение |
|--------|------------|
| **FSRS** | Free Spaced Repetition Scheduler — open-source алгоритм интервального повторения |
| **FIRe** | Fractional Implicit Repetition — модель оптимизации SRS для иерархических структур знаний |
| **SRS** | Spaced Repetition System — система интервального повторения |
| **Knowledge Graph** | Направленный ациклический граф (DAG) математических тем и их зависимостей |
| **Interleaving** | Чередование задач из разных тем в одной тренировочной сессии |
| **Mastery Level** | Уровень освоения темы (0.0–1.0), рассчитанный на основе FSRS-параметров карточек |
| **Retrievability** | Текущая вероятность успешного вспоминания (0–1), убывает со временем |
| **Stability** | Время в днях, за которое retrievability падает до 90% |
| **Due Card** | Карточка, у которой наступила дата следующего повторения |
| **Implicit Credit** | Неявное повторение базовой темы при решении продвинутой задачи (механизм FIRe) |
| **Calm UX** | Парадигма дизайна, минимизирующая тревожность и когнитивную нагрузку |
| **КИМ** | Контрольный измерительный материал — экзаменационный вариант ЕГЭ |
| **Кодификатор** | Официальный документ ФИПИ, описывающий все темы и навыки, проверяемые на ЕГЭ |
